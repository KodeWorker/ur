Findings tracked as GitHub issues in milestone **Phase 2 — Code Review Round 2**.

## Major

### [1] Zombie subprocess on timeout in `shell` tool
`bug` `performance`

`proc.kill()` sends SIGKILL but never awaits `proc.wait()`. The child process becomes a
zombie that persists until the event loop exits. In long-running `ur chat` sessions with
repeated timed-out commands this silently leaks OS resources.

**Location:** `src/ur/tools/builtin.py`, `shell()` timeout handler.

**Fix:**
```python
except TimeoutError:
    proc.kill()
    await proc.wait()
    return f"Error: command timed out after {timeout}s"
```

---

### [2] DNS rebinding not blocked by `_validate_url`
`security`

`_validate_url` correctly rejects raw IP literals that are private/loopback/link-local.
However any hostname that resolves to a private address (e.g. `169.254.169.254.nip.io`)
bypasses the guard entirely — `ipaddress.ip_address(host)` raises `ValueError` for
hostnames and the function returns early without resolving the name.

Complete mitigation requires a custom `httpx` transport or dual-resolution (resolve →
check → connect). For `browser_get`, Playwright makes its own DNS resolution which is
entirely outside `_validate_url`'s control.

**Location:** `src/ur/tools/builtin.py`, `_validate_url()` lines 28–33.

**Fix:** At minimum document the known limitation with a comment. Consider a partial
hostname denylist for common dynamic-DNS bypass patterns (`.nip.io`, `.xip.io`,
`.sslip.io`).

---

### [3] Rich markup injection in `TurnWidget` and `add_error`
`bug` `security`

`TurnWidget.compose` embeds `self._user_text` directly into a `markup=True` Static
without escaping. If the user types `[red]oops[/red]` or a malformed tag, Rich will
either render unintended markup or raise `MarkupError` and crash the widget.
The same issue appears in `add_error` where `str(e)` (which may contain
attacker-controlled content from a tool result) is embedded in a `markup=True` string.

**Location:** `src/ur/tui.py` — `TurnWidget.compose` and `add_error`.

**Fix:** Escape user / exception text before interpolation:
```python
from rich.markup import escape
# in compose:
f"[bold blue]you[/bold blue] › {escape(self._user_text)}"
# in add_error:
f"[red]Error:[/red] {escape(text)}"
```

Note: `ToolConfirmWidget` already uses `markup=False` for tool name and args — that
widget is not affected.


### [10] `_load_one` does not clean up `sys.modules` on `BaseException`
`bug`

The `sys.modules` cleanup after a failed `exec_module` is inside `except Exception`.
If a plugin's top-level code calls `sys.exit()` or raises `KeyboardInterrupt`
(`BaseException` but not `Exception`), the half-initialised module remains in
`sys.modules`.

**Location:** `src/ur/tools/plugins.py`, `_load_one`.

**Fix:** Broaden to `except BaseException` and re-raise non-`Exception` types after
cleanup:
```python
except BaseException as exc:
    logger.warning("Plugin %s: import failed — skipped", path, exc_info=True)
    sys.modules.pop(module_name, None)
    if not isinstance(exc, Exception):
        raise
    return
```

## Major

### [A] `http_get` follows redirects to private IPs — SSRF bypass

**Location:** `src/ur/tools/builtin.py` — `http_get`

`_validate_url` checks only the initial URL supplied by the model. `httpx.AsyncClient`
is called with `follow_redirects=True` (httpx default: 20 hops). A public server at
`https://example.com/` can issue a `301` to `http://192.168.1.1/` or
`http://169.254.169.254/latest/meta-data/`, and httpx will follow it without any
further validation. The entire SSRF defence is bypassed for every redirect hop after
the first.

**Fix:** Disable redirects (`follow_redirects=False`) and let the model decide whether
to follow by calling `http_get` again with the `Location` header value. This is the
simplest fix and fits the existing tool design. Add a note to the tool description that
redirects are not followed. Alternatively, add an `httpx` event hook that calls
`_validate_url` on every redirect target before following.

---

### [B] `_configure_logging` not idempotent — handlers accumulate on repeated calls

**Location:** `src/ur/main.py` — `_configure_logging`, `_settings`

`_settings()` calls `_configure_logging(s)` every time it is invoked. Although
`get_settings()` is `lru_cache`'d (same `Settings` object each time), a new
`RotatingFileHandler` is constructed and added to the root logger on every call.
The existing guard (`if not logging.root.level`) protects only the log level, not the
handler list. Any scripting wrapper, test that bypasses the patch, or future library
API that calls `_settings()` more than once in the same process will silently write
every log record multiple times.

**Fix:**
```python
def _configure_logging(settings: Settings) -> None:
    log_file = settings.logs_dir / "ur.log"
    if any(
        isinstance(h, logging.handlers.RotatingFileHandler)
        and getattr(h, "baseFilename", None) == str(log_file)
        for h in logging.root.handlers
    ):
        return
    handler = logging.handlers.RotatingFileHandler(...)
    ...
```

### [C] `max_iterations` exhaustion is silent — no feedback, session falsely `completed`

**Location:** `src/ur/agent/loop.py` — `run()`

When the model exhausts `max_iterations` by issuing repeated tool calls, the `for`
loop exits without an `else` branch. As a result:

1. No `StreamChunk` is yielded — the TUI shows only tool widgets with no conclusion.
2. `session.add_assistant_message()` is never called — the transcript ends with the
   last tool result.
3. `session.complete()` is still called by `_stream_turn`, so `ur history` records the
   session as `"completed"` even though no final reply was produced.

**Fix:** Add a `for…else` clause that yields a sentinel chunk and records a synthetic
assistant message:
```python
else:
    msg = f"(Agent reached the maximum of {max_iterations} iterations.)"
    yield StreamChunk(kind="content", text=msg)
    session.add_assistant_message(msg)
```

`test_run_loops_up_to_max_iterations` should be extended to assert that a content
chunk is yielded and that a final assistant message exists in `session.messages`.

### [F] `read_file` reads the entire file into memory before truncating to `max_lines`

**Location:** `src/ur/tools/builtin.py` — `read_file`

`await f.readlines()` buffers every line of the file regardless of `max_lines`. For a
100 MB log file with `max_lines=200` (the operator default), the function allocates
the full file in memory and discards 99.9 % of it.

**Fix:** Iterate line-by-line and stop early:
```python
async with aiofiles.open(resolved) as f:
    head: list[str] = []
    total = 0
    async for line in f:
        total += 1
        if total <= max_lines:
            head.append(line)
if total > max_lines:
    head.append(f"\n... ({total - max_lines} more lines)")
return "".join(head)
```

### [M1] Session message recorded before chunks yielded on tool denial

**Location:** `src/ur/agent/loop.py` — denial branch of `run()`

In the denial path, `session.add_tool_result_message(...)` is called *before* the
`tool_call` and `tool_result` chunks are yielded:

```python
# current (wrong order)
session.add_tool_result_message(tc_id, tc_name, result)
yield StreamChunk(kind="tool_call", text=tc_text)
yield StreamChunk(kind="tool_result", text=result)
```

The non-denied path (lines 72–75) correctly yields the `tool_call` chunk first, then
records the session message. If the generator is cancelled between the session write
and the `yield` statements, the session transcript will contain a tool-result message
whose corresponding tool-call chunk was never rendered — session and TUI diverge.

**Fix:** Mirror the non-denied path — yield both chunks first, then record the message:

```python
tc_text = f"{tc_name}({tc_args})"
yield StreamChunk(kind="tool_call", text=tc_text)
yield StreamChunk(kind="tool_result", text=result)
session.add_tool_result_message(tc_id, tc_name, result)
continue
```

### [M2] `_configure_logging` idempotency guard has no test

**Location:** `tests/unit/test_main.py`

Round 3 added an idempotency guard to `_configure_logging` but neither of the two
new tests calls it twice to verify the guard actually prevents handler accumulation.
A future refactor removing the early-return check would go undetected.

**Fix:** Add a test that calls `_configure_logging` twice and asserts exactly one
`RotatingFileHandler` is registered:

```python
def test_configure_logging_is_idempotent(tmp_settings: Settings) -> None:
    tmp_settings.ensure_dirs()
    root = logging.getLogger()
    before = list(root.handlers)
    new_handlers: list[logging.Handler] = []
    try:
        _configure_logging(tmp_settings)
        _configure_logging(tmp_settings)  # second call — must be a no-op
        new_handlers = [h for h in root.handlers if h not in before]
        rfh = [h for h in new_handlers
               if isinstance(h, logging.handlers.RotatingFileHandler)]
        assert len(rfh) == 1
    finally:
        for h in new_handlers:
            h.close()
            root.removeHandler(h)
```

## Suggestions

### [s1] Plugin module names collide across multiple directories

**Location:** `src/ur/tools/plugins.py` — `_load_one`

Module names are generated as `f"ur._plugin.{path.stem}"`. If plugins are ever
loaded from multiple directories, two files named `utils.py` from different dirs
would produce the same key in `sys.modules`, and the second plugin's code would
never execute. Not a current bug — only one directory is scanned — but a latent
hazard.

**Fix:** Document the single-directory assumption, or make the name path-unique:

```python
safe_dir = str(path.parent).replace("/", "_").replace(".", "_")
module_name = f"ur._plugin.{safe_dir}_{path.stem}"
```

### [s2] `web_search` uses `functools.partial` instead of a lambda

**Location:** `src/ur/tools/builtin.py` — `create_default_registry`

Unlike `read_file` (converted to a lambda so the model cannot override `max_lines`),
`web_search` uses `functools.partial(web_search, max_results=max_search_results)`.
The schema does not expose `max_results`, so the model cannot bypass the limit, but
a plugin author calling `registry.get("web_search").fn(...)` directly can. Convert
to a lambda for consistency:

```python
fn=lambda query: web_search(query, max_results=max_search_results),
```
