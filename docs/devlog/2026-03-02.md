# Textual TUI Rewrite ‚Äî Implementation Log

## Overview

The `run` and `chat` commands previously used `rich.Live` to stream LLM output, displaying
reasoning as flat dim text with no interactivity. The goal was to make reasoning blocks
collapsible so users can toggle them open/closed.

**Discovery:** Rich 14.3.3 has no `Collapsible` widget and cannot do interactive toggling.
Textual (already available in the venv at 8.0.1) has a `Collapsible` widget and a full TUI
framework. The two commands were rewritten using Textual. `history` and `init` remain
unchanged (still use Rich).

## Implementation

### `src/ur/tui.py` ‚Äî New file

**`TurnWidget(Vertical)`** ‚Äî one user‚Üíagent conversational turn.

Composes statically: only the user-label `Static`. All other children are mounted
dynamically as chunks arrive from the streaming worker:

```
append_reasoning(text)     ‚Äî async; mounts a new Collapsible on first call per segment,
                             then updates its Static in-place with accumulated text
reset_reasoning()          ‚Äî sync; closes the current segment so the next reasoning chunk
                             opens a fresh Collapsible
append_content(text)       ‚Äî mounts fresh Markdown on first call; updates it thereafter
add_tool_call(text)        ‚Äî resets _active_content, mounts dim ‚öô Static
add_tool_result(text)      ‚Äî mounts dim ü§ñ preview Static (first line, ‚â§120 chars)
add_error(text)            ‚Äî resets _active_content, mounts red error Static
```

Two independent reset mechanisms keep visual ordering correct:

- **`_active_content`** ‚Äî reset by `add_tool_call`: each content segment after a tool
  round-trip gets its own `Markdown`.
- **`_active_reasoning`** ‚Äî reset by `reset_reasoning()`: each reasoning segment
  (before the first tool call, between tool calls, after the last) gets its own
  `Collapsible`, independently toggle-able by the user.

Visual sequence for a reasoning ‚Üí tool ‚Üí reasoning ‚Üí reply turn:

```
you ‚Ä∫ <task>
‚ñ∂ reasoning          ‚Üê collapsed, toggleable  (segment 1: pre-tool)
‚öô tool_name(args)
ü§ñ result preview
‚ñ∂ reasoning          ‚Üê collapsed, toggleable  (segment 2: post-tool)
<reply markdown>
```

**`UrApp(App[None])`** ‚Äî dual-mode TUI (run / chat).

Layout:
```
Header (title="ur", sub_title="<model>  <session[:8]>")
ScrollableContainer id="scroll"   [height: 1fr]
    TurnWidget(s) mounted dynamically
Input id="input-bar"              [chat mode only]
Static id="status"                [token usage / provider hint]
Footer
```

Key behaviours:
- **run mode:** `on_mount` immediately calls `_start_turn(task)` then the app exits after the
  turn completes
- **chat mode:** `on_mount` focuses the Input; each submitted message triggers a new turn;
  Input is disabled while the worker runs and re-enabled after
- **`@work(exclusive=True) _stream_turn(turn)`:** consumes `agent_run` async generator,
  routes each `StreamChunk` to the appropriate `TurnWidget` method, then:
  - `session.complete()` (run mode only)
  - `session.fail()` + error display on `Exception`
  - `session.interrupt()` + re-raise on `BaseException`
  - `save_session` + status-bar update in `finally` (always runs)
- **`action_request_quit`** (Ctrl+C / Ctrl+D): checks whether the streaming worker is still
  running; if so ‚Üí `session.interrupt()`; if idle ‚Üí `session.complete()`; saves and exits
- **`action_quit`** override: Textual's command palette "Quit" calls `action_quit` rather
  than `action_request_quit`; overridden to delegate so quit always goes through the same
  lifecycle path

**`_make_registry(no_tools, settings)`** ‚Äî mirrors the helper removed from `main.py`.
Graceful `ImportError` fallback if `[tools]` extra is not installed.

**`launch_run` / `launch_chat`** ‚Äî public async entry points called by `main.py`.

## Post-Implementation Features

### Per-segment reasoning collapsibles

**Request:** Show separate `<reasoning #1>` / `<Tool Call>` / `<Tool Result>` /
`<reasoning #2>` blocks rather than a single merged reasoning toggle per turn.

**Changes:**

`TurnWidget`:
- Removed the static `Collapsible` from `compose()` ‚Äî only the user-label is now static.
- Added `_active_reasoning: Static | None = None` to `__init__`.
- `append_reasoning` made `async`: mounts a new `Collapsible(Static(...), title="reasoning",
  collapsed=True)` when `_active_reasoning is None`, otherwise updates in-place.
- New `reset_reasoning()` (sync): sets `_active_reasoning = None` to end the segment.

`_stream_turn`:
- `await turn.append_reasoning(...)` (now async).
- Calls `turn.reset_reasoning()` on `content` chunk (if reasoning was active) and
  unconditionally on `tool_call` chunk.

Test:
- `test_turn_widget_append_reasoning_updates_static`: changed sync call to `await`.

---

### Tool call confirmation

**Request:** Before executing a tool, ask the user to allow or deny it inline (not a modal
pop-up), with an optional denial reason passed back to the model.

**Changes:**

`agent/loop.py`:
- Added `confirm_tool: Callable[[str, str], Awaitable[str | None]] | None = None` to `run()`.
- Called between `yield StreamChunk(kind="tool_call")` and `_execute_tool`; if `denial is
  not None`, records a tool-result message and skips execution:
  - `denial == ""` ‚Üí `"Tool call denied by user."`
  - `denial != ""` ‚Üí `f"Tool call denied by user with {denial}"`

`tui.py` ‚Äî new `ToolConfirmWidget(Vertical)`:
- Mounted inline in `#scroll` (not a modal screen).
- Composes: bold warning tool name, dim args preview (‚â§300 chars), prompt line, `Input`.
- `on_mount`: creates `asyncio.Future`; focuses the Input.
- `on_input_submitted`: calls `event.stop()` (prevents bubbling to `UrApp.on_input_submitted`);
  resolves the future ‚Äî `None` if allowed, `reason` (possibly `""`) if denied; replaces
  prompt+input with a `‚úì allowed` or `‚úó denied` label.
- `wait_for_response()`: awaited by the `_confirm_tool` closure in `_stream_turn`.

Allow logic (strict): only `"yes"` or `"YES"` (exact match) ‚Üí allowed. Empty input and
everything else ‚Üí denied.

`_stream_turn` closure:
```python
async def _confirm_tool(name: str, args: str) -> str | None:
    widget = ToolConfirmWidget(name, args)
    await scroll.mount(widget)
    scroll.scroll_end(animate=False)
    result = await widget.wait_for_response()
    scroll.scroll_end(animate=False)
    return result
```

Visual appearance:
```
‚îÉ ‚öô shell                    ‚Üê bold warning colour, left border
‚îÉ {"cmd": "rm -rf /"}        ‚Üê dim args preview
‚îÉ
‚îÉ Allow? yes/YES to allow ‚Äî ‚Üµ or type a reason + ‚Üµ to deny
‚îÉ [reason for denial (‚Üµ to deny)‚Ä¶_____________]
```

## Bugs Found and Fixed

### Bug 1 ‚Äî `App._registry` name collision (critical)

**Symptom:** `NoMatches` on `query_one("#scroll")`, then `TypeError: 'NoneType' object is
not iterable` in `_close_all()` during shutdown.

**Root cause:** Textual's `App` has an internal attribute `_registry: WeakSet[DOMNode]` that
tracks all mounted widgets. Our `UrApp.__init__` assigned `self._registry = registry`
(a `ToolRegistry | None`), silently overwriting it with `None`.

**Fix:** Renamed the instance attribute to `self._tool_registry` everywhere.

---

### Bug 2 ‚Äî `App._task` name collision (mypy)

**Symptom:** mypy error ‚Äî "Incompatible types in assignment (expression has type `str | None`,
variable has type `Task[Any] | None`)" on `self._task = task`.

**Root cause:** Textual's `App` has an internal attribute `_task: Task[Any] | None`.
Assigning our `task: str | None` parameter to `self._task` triggered the type mismatch.

**Fix:** Renamed to `self._run_task` everywhere in `tui.py`.

---

### Bug 3 ‚Äî `session.complete()` called in chat mode

**Symptom:** In chat mode, after a turn completed, the session status became `"completed"`
instead of staying `"running"`, causing the Input re-enable path to behave incorrectly and
making lifecycle tests fail.

**Root cause:** `_stream_turn` called `self._session.complete()` unconditionally after the
agent loop finished, regardless of mode.

**Fix:** Wrapped with `if self._mode == "run": self._session.complete()`. Chat sessions
stay `"running"` between turns; only `action_request_quit` (Ctrl+C) transitions them to
`"interrupted"`.

---

### Bug 4 ‚Äî `Static.renderable` does not exist in Textual 8

**Symptom:** `AttributeError: 'Static' object has no attribute 'renderable'` in the
reasoning test.

**Root cause:** The test used `str(static.renderable)` to read a Static widget's content,
but Textual 8 stores it as the name-mangled `_Static__content`.

**Fix:** Changed assertion to `str(static._Static__content)`.

---

### Bug 5 ‚Äî `ctrl+d` binding does not fire in headless `auto_pilot` tests

**Symptom:** `assert session.status == "interrupted"` failed ‚Äî status remained `"running"`
after `await pilot.press("ctrl+d")`.

**Root cause:** `ctrl+d` (EOF) is intercepted at a lower level in headless mode and never
reaches Textual's binding system. `ctrl+c` fires correctly (confirmed by passing run-mode test).

**Fix:** Changed both chat-mode autopilot callbacks to `await pilot.press("ctrl+c")`.

---

### Bug 6 ‚Äî Sessions always showed "interrupted" even after LLM finished

**Symptom:** `ur history` showed every session as `interrupted`, even those where the model
finished streaming normally and the user then pressed Ctrl+C to exit.

**Root cause:** `action_request_quit` called `session.interrupt()` unconditionally whenever
`session.status == "running"`, with no distinction between "worker still streaming" and
"worker already finished, user exiting cleanly".

**Fix:** Check `any(w.is_running for w in self.workers)`:
- Worker running ‚Üí `session.interrupt()` (user cut it short)
- Worker idle ‚Üí `session.complete()` (conversation ended normally before quit)

---

### Bug 7 ‚Äî Command palette "Quit" left session status as "running"

**Symptom:** Quitting via the command palette (not Ctrl+C) skipped `action_request_quit`
and exited without persisting the session or updating its status.

**Root cause:** Textual dispatches "Quit" from the command palette (and the default `q`
binding) to `action_quit`, not `action_request_quit`. `UrApp` had no override for
`action_quit`, so Textual's default implementation ran, calling `self.exit()` directly.

**Fix:** Added `action_quit` override that simply delegates to `action_request_quit`.

---

### Bug 8 ‚Äî Input widget visual glitch (double border)

**Symptom:** A visible top-border line appeared above the `#input-bar` Input, distinct from
the Input widget's own Textual border, creating a double-line effect.

**Root cause:** The `#input-bar` CSS block included `border-top: solid $primary`, which
drew a container border that overlapped with Textual's own Input border rendering.

**Fix:** Removed the `border-top` line from the `#input-bar` CSS block.

---

### Bug 9 ‚Äî `pilot.pause()` between sequential mounts causes `MountError` (original Bug 6)

**Symptom:** `MountError: Can't mount widget(s) before TurnWidget() is mounted` on the
second `append_content` call in `test_turn_widget_tool_call_resets_content_segment`.

**Root cause:** Adding `await pilot.pause()` between `add_tool_call` and the subsequent
`append_content` allowed Textual's event loop to run, which processed some internal event
that left the `TurnWidget` in a state where `is_attached` returned `False`.

**Fix:** Removed the extra `pilot.pause()`. Sequential `await widget.method()` calls that
each internally call `await self.mount(...)` must not have pauses inserted between them ‚Äî
let them run back-to-back and add a single `pilot.pause()` only after the sequence is done.

# Phase 2 Tool Suite

## Overview

Extended the built-in tool set beyond `shell`, `read_file`, and `write_file` with four new
capabilities: headless browser, raw HTTP, web search, and a plugin loader for user-defined tools.
All operator-facing configuration (timeouts, limits) was moved out of the LLM-visible schemas
and into `Settings` fields, configurable via `UR_*` env vars.

---

## `browser_get` ‚Äî Playwright + markdownify

Added `browser_get(url)` to `src/ur/tools/builtin.py`. Navigates headlessly with Playwright
(`wait_until="domcontentloaded"`) and converts the rendered HTML to markdown via `markdownify`,
stripping `script`, `style`, and `head` tags. Returns markdown with links preserved ‚Äî ideal
for agent link-following at near-minimum token cost.

Dependencies added to `[tools]` extra: `playwright>=1.45`, `markdownify>=0.13`.

```
# after install:
uv run playwright install chromium
```

Lazy imports (`from playwright.async_api import async_playwright`) inside the function body
keep the `[tools]` extra optional. The existing `ignore_missing_imports = true` mypy override
on `ur.tools.builtin` covers all optional imports.

**Tests (3):** mocked via `mocker.patch("playwright.async_api.async_playwright", ...)` ‚Äî
returns markdown, truncates long output, surfaces errors as `"Error: ..."` strings.

---

## `web_search` ‚Äî DuckDuckGo via `ddgs`

Added `web_search(query, max_results)` using `DDGS().text()` from the `ddgs` package (the
`duckduckgo-search` package was renamed; old `ddg()` function no longer exists). Returns
numbered results with title, URL, and snippet.

Dependency: `ddgs>=9.0` (replaced `duckduckgo-search>=6.0` in `[tools]` extra).

**Tool description design:** descriptions explicitly guide the model on when to use each
web tool:
- `web_search` ‚Äî "when you do not already have a URL ‚Ä¶ follow up with http_get or browser_get"
- `http_get` ‚Äî "Use when you already have a URL and the page does not require JavaScript"
- `browser_get` ‚Äî "Use when you already have a URL and the page requires JavaScript"

**Tests (3):** patched at class level (`mocker.patch("ddgs.DDGS", return_value=mock_instance)`)
so `DDGS().text()` returns controlled data without a network call.

---

## Plugin loader ‚Äî `src/ur/tools/plugins.py`

New module allowing users to drop `.py` files into `~/.local/share/ur/tools/` (or
`$UR_DATA_DIR/tools/`) to register custom tools at startup.

```python
def load_plugins(registry: ToolRegistry, tools_dir: Path) -> None: ...
def _load_one(registry: ToolRegistry, path: Path) -> None: ...
```

Each plugin must export a `register(registry: ToolRegistry) -> None` function.
Plugins are loaded alphabetically; last writer wins on name collision, so plugins
can override built-ins.

Three independent `try/except` guards in `_load_one` ensure a bad plugin never
crashes the registry load: import errors, missing `register`, and `register()` raising
are all silently skipped.

`tools_dir` is a `@property` on `Settings` (`data_dir / "tools"`), matching the pattern
used by `db_path`, `workspaces_dir`, and `logs_dir`. `ensure_dirs()` creates it on first use.

**Tests (7):** missing dir, valid plugin, syntax error, missing register fn, raising register,
alphabetical last-writer-wins, plugin overrides a built-in.

---

## Per-session workspace directory

Each `ur run` / `ur chat` session gets an isolated directory at
`$UR_DATA_DIR/workspaces/<session-id>/`. Shell commands and file tools resolve relative paths
against this directory so the agent works naturally with relative paths out of the box.

**Changes:**

- `shell`, `read_file`, `write_file` gained a `cwd: Path | None = None` parameter.
  `shell` passes it to `asyncio.create_subprocess_shell(cwd=...)`.
  `read_file` / `write_file` resolve relative paths via `(cwd / p) if not p.is_absolute()`.
- `create_default_registry` gained `workspace_dir: Path | None = None`; bound into the three
  file/shell tools via `functools.partial`.
- `launch_run` / `launch_chat` in `tui.py` create the workspace dir before building the
  registry and pass it through `_make_registry` ‚Üí `UrApp`.

No system-prompt injection was needed ‚Äî relative paths just work because `cwd` is set.

---

## Schema design ‚Äî operator config out of LLM-visible schemas

Established principle: the LLM tool schema should only expose parameters with semantic meaning
to the model. Operator configuration (limits, timeouts) belongs in `Settings`, bound at
registry-build time via `functools.partial`.

**Removed from schemas:**
- `timeout` from `shell`, `http_get`, `browser_get`
- `max_results` from `web_search`

**Kept in schema:**
- `max_lines` on `read_file` (semantic ‚Äî LLM chooses whether to read the full file or just
  the head)

# Code Review ‚Äî Phase 2 Round 1

Automated code review of the Phase 2 tool suite. Findings tracked as GitHub issues in
milestone **Phase 2 ‚Äî Code Review Round 1**.

## Critical

### [#68] Path traversal in `read_file`/`write_file` ‚Äî workspace sandbox is illusory
`bug` `security`

`read_file` and `write_file` resolve relative paths against `cwd` (the per-session workspace)
but never verify the resolved path stays inside it. `path="../../../etc/passwd"` succeeds
today. `write_file` becomes a write-anywhere primitive; `read_file` can exfiltrate any
readable file.

**Location:** `src/ur/tools/builtin.py` lines 49, 66.

**Fix:** After resolving, assert containment with `resolved.relative_to(cwd.resolve())`;
return an error string if it raises `ValueError`.

---

### [#69] `max_lines` operator limit bypassed by model via JSON schema
`bug` `security`

`max_lines` appears in the `read_file` JSON schema, so the model can pass its own value and
override the `functools.partial` binding ‚Äî defeating the Phase 2 design goal of keeping
operator config out of LLM-visible schemas. `functools.partial` only sets a default; kwarg
dispatch from `_execute_tool` overrides it.

**Location:** `src/ur/tools/builtin.py` lines 178‚Äì185.

**Fix:** Remove `max_lines` from the schema and enforce the operator limit in a wrapper
that discards any model-supplied value.

---

## Major

### [#70] `web_search` blocks the event loop ‚Äî `DDGS().text()` is synchronous
`bug` `performance`

`DDGS().text()` performs blocking HTTP I/O on the calling thread. Wrapping it in `list()`
does not make it async. This will freeze the Textual TUI for the duration of every search.
All other built-in tools are genuinely async.

**Location:** `src/ur/tools/builtin.py` line 123.

**Fix:** `await loop.run_in_executor(None, lambda: list(DDGS().text(...)))`.

---

### [#71] SSRF ‚Äî `http_get` and `browser_get` accept arbitrary URLs
`security`

No URL scheme or address validation. An agent or injected prompt can target internal
endpoints (`http://localhost:11434/`, `http://169.254.169.254/`). `http_get` follows
redirects (`follow_redirects=True`), compounding the risk.

**Location:** `src/ur/tools/builtin.py` lines 74‚Äì115.

**Resolution needed:** either add scheme + RFC 1918/loopback blocking, or document explicitly
as an accepted risk for a local developer tool.

---

### [#72] `browser_get` launches a new Chromium process per call (1‚Äì3 s cold-start)
`performance`

`async with async_playwright() as p: browser = await p.chromium.launch(...)` runs on every
call. In multi-step agent tasks visiting several pages this creates a significant serial
bottleneck. Documented as a known v1 limitation; a module-level lazy browser singleton is
the follow-up fix.

**Location:** `src/ur/tools/builtin.py` lines 79‚Äì97.

---

## Minor

### [#73] Replace `assert` with `RuntimeError` in `ToolConfirmWidget.wait_for_response`
`code-quality`

`assert self._future is not None` is stripped by `-O`, producing an unhelpful `AttributeError`
on `None` instead of a clear message.

**Location:** `src/ur/tui.py` line 234.

`bug`

### [#76] Document plugin security model ‚Äî `tools_dir` executes arbitrary code
`documentation` `security`

Any `.py` file in `~/.local/share/ur/tools/` is executed unconditionally at startup with
the full privileges of the running user. Neither the module docstring nor the README warns
operators that only trusted files should be placed there.

**Location:** `src/ur/tools/plugins.py`, README.
