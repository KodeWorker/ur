# Fix: Single Ctrl+C exit from `_chat` and correct history status

Two related bugs in `_chat` (both in `src/ur/main.py`):

## Root cause

On Python 3.11+, `asyncio.run()` replaces `signal.default_int_handler` with its own
`_cancel_main_task` handler, which calls `task.cancel()` and returns — it **never** raises
`KeyboardInterrupt`.  This breaks two code paths:

| # | Symptom | Trigger |
|---|---|---|
| A | Double Ctrl+C required to quit at `you ›` prompt | `_cancel_main_task` only schedules cancellation; blocking `console.input()` never sees `KeyboardInterrupt` on the first press |
| B | `ur history` shows sessions as `running` after Ctrl+C during streaming | `asyncio.CancelledError` (a `BaseException`, not `Exception`) bypassed both `except` handlers; `session.interrupt()` was never called, so every `finally`-save recorded `"running"` |

## Fix A — prompt (L130-L144)

| File | What changed |
|---|---|
| `src/ur/main.py` | Added `import signal`; wrap `console.input()` with `signal.signal(signal.SIGINT, signal.default_int_handler)` / `finally: signal.signal(signal.SIGINT, _prev_sigint)` so the default handler (which raises `KeyboardInterrupt`) is active for the duration of the blocking call; asyncio's handler is restored immediately after |

## Fix B — streaming (L169-L185)

| File | What changed |
|---|---|
| `src/ur/main.py` | Added `except BaseException: session.interrupt(); raise` between `except Exception` and `finally` in the inner streaming try-block; catches `asyncio.CancelledError`, marks the session interrupted, re-raises so asyncio task cleanup proceeds normally; the existing `finally` then saves with `"interrupted"` status |

## Finding descriptions

**#59 — `_run` missing `BaseException` handler (`src/ur/main.py` ~L74)**

`_chat`'s inner streaming block has `except BaseException: session.interrupt(); raise` (added in Round 4). `_run` has no equivalent. When `asyncio.CancelledError` is raised during streaming (e.g., user presses Ctrl+C), the `except Exception` handler does not fire, the `finally` saves with `status="running"`, and the session appears as `"running"` in history indefinitely. Fix: add the same clause between `except Exception` and `finally` in `_run`.

**#60 — `max_iterations` validation missing (`src/ur/config.py` ~L20)**

`max_iterations: int = Field(default=20)` has no lower-bound constraint. `UR_MAX_ITERATIONS=0` silently causes `loop.run()` to yield nothing — the LLM is never called, no assistant message is appended, and the session is saved with only the user message. Fix: `Field(default=20, ge=1)` so pydantic-settings raises a `ValidationError` at startup with a clear message.

**#62 — Missing DB index on `messages.session_id` (`src/ur/memory/db.py` ~L24)**

The `messages` table has no index on `session_id` despite it being the only filter used by `get_session_messages`. Every call is a full table scan. Fix: add `CREATE INDEX IF NOT EXISTS idx_messages_session_id ON messages(session_id);` to the schema string.

**#63 — Dead `task[:80]` slice in `_history` (`src/ur/main.py` ~L269)**

`table.add_column("Task", max_width=52, no_wrap=True)` causes Rich to truncate at 52 characters. The Python-side slice `s["task"][:80]` never has any effect. Fix: remove the slice and let Rich handle truncation, or change it to `[:52]` to match the column width.

**#64 — No test for `max_iterations=0` (`tests/unit/test_loop.py`)**

`test_run_stops_after_one_iteration_in_phase1` covers the break-after-one contract but there is no test for zero iterations. When `max_iterations=0` the generator yields nothing and no assistant message is added, silently producing a broken session. A test would document whether this is intentional or should be an error.

**#65 — No test for `_run` `BaseException` path (`tests/`)**

There is no test that runs `_run` end-to-end and asserts that a `CancelledError` or `KeyboardInterrupt` during streaming results in `session.status == "interrupted"`. Without it, the bug in #59 would not be caught by CI after being fixed.

**#66 — Zero CLI coverage (`tests/`)**

No tests exercise `_run`, `_chat`, or `_history`. All 65 existing tests cover the layer below the CLI. Bugs in the wiring (such as #59) go undetected. Suggested additions at minimum: `_run` happy path (assert `session.complete()` and `save_session` called); `_run` exception path (assert `session.fail()` and `typer.Exit(1)` raised); `_history` list path; `_history` prefix lookup.
