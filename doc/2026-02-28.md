# Fix: Single Ctrl+C exit from `_chat` and correct history status

Two related bugs in `_chat` (both in `src/ur/main.py`):

## Root cause

On Python 3.11+, `asyncio.run()` replaces `signal.default_int_handler` with its own
`_cancel_main_task` handler, which calls `task.cancel()` and returns — it **never** raises
`KeyboardInterrupt`.  This breaks two code paths:

| # | Symptom | Trigger |
|---|---|---|
| A | Double Ctrl+C required to quit at `you ›` prompt | `_cancel_main_task` only schedules cancellation; blocking `console.input()` never sees `KeyboardInterrupt` on the first press |
| B | `ur history` shows sessions as `running` after Ctrl+C during streaming | `asyncio.CancelledError` (a `BaseException`, not `Exception`) bypassed both `except` handlers; `session.interrupt()` was never called, so every `finally`-save recorded `"running"` |

## Fix A — prompt (L130-L144)

| File | What changed |
|---|---|
| `src/ur/main.py` | Added `import signal`; wrap `console.input()` with `signal.signal(signal.SIGINT, signal.default_int_handler)` / `finally: signal.signal(signal.SIGINT, _prev_sigint)` so the default handler (which raises `KeyboardInterrupt`) is active for the duration of the blocking call; asyncio's handler is restored immediately after |

## Fix B — streaming (L169-L185)

| File | What changed |
|---|---|
| `src/ur/main.py` | Added `except BaseException: session.interrupt(); raise` between `except Exception` and `finally` in the inner streaming try-block; catches `asyncio.CancelledError`, marks the session interrupted, re-raises so asyncio task cleanup proceeds normally; the existing `finally` then saves with `"interrupted"` status |
