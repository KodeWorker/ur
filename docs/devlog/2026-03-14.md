# 2026-03-14

## Phase 3 — Persona injection, `ur clean --persona`, and TUI cleanup

### Removed `initial_system_prompt` from `make_tui` / `FtxuiTui`

`--system-prompt` flag was dropped from `ur chat` (use `/load-prompt` instead).
`FtxuiTui` constructor and `make_tui()` factory no longer take an
`initial_system_prompt` parameter. Call site in `cmd_chat` updated accordingly.

`kSlashCommands` autocomplete list updated to include all current commands:
`/help`, `/exit`, `/compact`, `/clear`, `/title`, `/persona`,
`/load-prompt`, `/save-prompt`.

### Persona injection into chat context

Before each `provider.stream()` call, `Chat::run()` now builds an
`effective_system` string that prepends known persona facts to the user's
system prompt:

```
Known facts about the user:
- key: value
...

<user system prompt>
```

The number of facts included is controlled by `UR_NUM_PERSONA` (env var):
`0` or unset = all facts; `N > 0` = latest N facts ordered by `updated_at DESC`.

### `select_persona` — recency order with optional SQL-level limit

`Database::select_persona(size_t limit = 0)` now uses:

```sql
SELECT key, value, updated_at FROM persona ORDER BY updated_at DESC [LIMIT N]
```

Sorting and limiting done in SQL rather than in C++. API unchanged for callers
that pass no argument (returns all facts, most-recently-updated first).

### `PersonaUpdater` — full conversation context, filtered extraction input

`maybe_update` now receives the full `std::vector<Message>& context` instead of
a single `user_msg` / `assistant_msg` pair.

`assistant_msg` parameter removed from `maybe_update` signature (`.hpp` + `.cpp`).

**Problem**: passing all assistant messages to the extractor caused assistant-generated
content (explanations, suggestions) to be attributed as facts about the user.

**Fix** (option 2 — preceding-assistant-only): for each user turn, the extraction
input includes only the single assistant message immediately before it (Q&A context),
then the user reply. All other assistant content is dropped. The system prompt is also
tightened to explicitly ignore assistant-stated content:

```
For each user turn i:
  if context[i-1].role == "assistant":  emit [Assistant]: ...
  emit [User]: ...
```

System prompt updated to: *"Only extract facts the user has explicitly stated or
clearly implied about themselves — ignore anything the assistant said."*

### `/persona` slash command — force update and display facts

`/persona` now calls `maybe_update` with `force_update=true` (bypasses the
`is_meaningful` guard that blocks extraction on short conversations), then
immediately displays the current persona facts from the database:

```
🪞persona updated
key: value
...
```

Shows `"🪞no persona facts yet"` if the table is empty.

### `ur clean --persona`

New flag added to `cmd_clean`. Drops only the `persona` table; sessions and
messages are untouched. `Database::drop_persona()` added (`.hpp` + `.cpp`).

Usage string and README updated:
```
ur clean [--database|--workspace|--persona]
```

## Design decisions

### Phase 4 — `run_agent` built-in tool dropped

`run_agent` (spawn `ur run` as a subprocess) was removed from the Phase 4 plan.
Rationale: `bash` already subsumes it — if bash is enabled the agent can call
`ur run` directly. The depth-limiting logic (`UR_AGENT_DEPTH`) and subprocess
output capture added complexity for a niche multi-agent pattern most users won't
need. Multi-agent delegation is deferred to Phase 7 where it can be designed
properly.

### Phase 4 — built-in tools use `Tool` ABC, compiled into `ur_lib`

Considered building all tools (including built-ins) as `.so`/`.dll` loaded at
runtime for a uniform plugin interface. Rejected: single-binary deployment is
lost and missing `.so` files would silently break core functionality. Decision:
built-ins compiled into `ur_lib`, registered before external plugins are scanned.
External plugins with the same name override built-ins (same `Tool` ABC contract
throughout). Users can replace any built-in by dropping a same-named `.so` into
`$root/tools/`.

### Phase 4 — `read_image` built-in added (disabled by default)

`read_image` validates and returns an image path to vision-capable models.
Path encoding is left to llama.cpp — `ur` only validates the path and passes it
through. Disabled by default in `tools.json`; enable only when the active model
supports vision.

### Phase 7 — Code agent planned (post Phase 6)

Rather than one JSON tool call per LLM turn (Phase 4), Phase 7 introduces a
code-agent mode: the LLM generates a Python script that orchestrates multiple
tool calls with real control flow (loops, conditionals, error handling) in a
single turn. Requires Phase 6 Docker sandbox for safe execution.

Key decisions recorded in `docs/plan/phase7.md`:
- Python tool stubs auto-generated from the `Tool` ABC registry — no tools need porting
- Tool side-effects (file I/O, web fetch) stay on the host, routed through the
  same `SandboxPolicy` as Phase 4; Docker only runs the orchestration script
- Fallback to JSON tool-calling loop when the model emits no Python block
- `"modes": ["json", "code"]` field in `tools.json` for per-tool opt-in/out

## Bug fixes — code review issues

### #182 — data race on `system_prompt` (analysis, fix pending)

`d->system_prompt` in `FtxuiTui::Impl` is a plain `std::string` accessed from
two threads simultaneously: the ftxui event loop thread writes to it on every
keystroke (via the `ftxui::Input` binding), while the chat loop thread reads it
via `tui.system_prompt()` before each `provider.stream()` call. Fix: guard the
field with a `std::mutex`; `system_prompt()` and `set_system_prompt()` acquire
the lock. The `ftxui::Input` binding must be changed to write through the
protected accessor rather than directly to the raw string.

### #184 — `--system-prompt` inconsistency closed

Issue was about `--system-prompt` behaving differently between `cmd_run` and
`cmd_chat`. Moot: `--system-prompt` was dropped from `ur chat` (use
`/load-prompt` instead). Closed as resolved-by-design.

### #185 — `env_int` error reporting via `std::cerr`

`env_int` in `env.cpp` previously caught `std::stoi` exceptions silently and
returned the fallback. Now prints a warning to `std::cerr`:

```
ur: warning: VAR_NAME="bad_value" is not a valid integer, using <fallback>
```

Decision: `Logger&` injection considered (via `HttpProvider`) but rejected —
`env_int` is a generic utility called from multiple sites, and tying it to
`Logger` would couple a low-level utility to a higher-level abstraction and
create a chicken-and-egg problem at startup. `std::cerr` keeps `env.cpp`
fully independent. `<iostream>` added to includes.

### Spinner UX — confirmed correct behaviour

Clarified that the spinner stopping on the first streaming token (rather than
at end-of-stream) is the intended design. The spinner signals TTFT latency
("waiting for anything"). Once tokens arrive the user has visible progress;
a spinner during streaming would be redundant and visually competing with
content. No change made.
