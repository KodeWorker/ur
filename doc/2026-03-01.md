# Phase 2: Tool Dispatch — Implementation Log

## Overview

Phase 2 gives the agent the ability to call tools (functions) and loop until the LLM stops
requesting tool calls. All 9 planned items were implemented plus several unplanned fixes
discovered during real-model testing with `ollama/qwen3:30b-custom`.

No DB schema changes were needed: `tool_calls TEXT`, `tool_call_id TEXT`, and `name TEXT`
columns already existed in the `messages` table.

---

## Planned Items (all completed)

### Item 1 — Tool Registry ✅

`src/ur/tools/registry.py` + `src/ur/tools/__init__.py`

`ToolRegistry` maps `name → ToolSpec(fn, schema)`. Exposes `as_tools_list()` in OpenAI
function-calling format. Plain object, not a singleton. No `@tool` decorator (explicit
schema dict passed to `register()` instead — simpler and provider-agnostic).

### Item 2 — Accumulate Tool Calls in `CompletionStream` ✅

`src/ur/llm/client.py` — `_iter()` accumulates `delta.tool_calls` fragments by index into
`tool_calls_acc`, exposing `self.tool_calls` and `has_tool_calls: bool` after the stream.

### Item 3 — Pass `tools` to `LLMClient.stream()` ✅

`stream(messages, tools=None)` — additive, all existing callers unaffected.

### Item 4 — Agentic Dispatch Loop ✅

`src/ur/agent/loop.py` — `run()` gains `registry: ToolRegistry | None`. When `registry`
is provided, dispatches tool calls and loops; `registry=None` preserves Phase 1 behaviour.
Private `_execute_tool()` wraps execution in `try/except Exception` — tool errors become
error strings, never propagate.

### Item 5 — `AgentSession` Updates ✅

`add_assistant_tool_call_message` takes `list[dict]` directly. New method
`add_tool_result_message(tool_call_id, name, content)`.

### Item 6 — Built-in Tools ✅

`src/ur/tools/builtin.py`: `shell`, `read_file`, `write_file`, `http_get`.
`create_default_registry(truncate_at, max_lines)` factory wires them up.

### Item 7 — Wire Registry into CLI ✅

`_make_registry(no_tools, settings)` in `main.py`. `--no-tools` flag on `run` and `chat`.
Graceful `ImportError` fallback (now with a user-visible warning).

### Item 8 — Tests ✅

`tests/conftest.py` — `make_tool_call_chunk`. `test_loop.py` — 6 new tool-dispatch cases.
`test_session.py` — updated call sites + `test_add_tool_result_message`.
`tests/unit/test_tools.py` — new: registry and builtin tests (builtin skipped without
`[tools]` extra via `pytest.importorskip`).

### Item 9 — History Display for Tool Messages ✅

`ur history <id>` renders `tool`-role messages in yellow with `tool_call_id` / `name`
metadata, and assistant messages with `tool_calls` show each call's name and arguments.

## Unplanned Fixes (discovered during testing)

### Fix A — Tool limits configurable via env

`UR_TOOL_BUILTIN_TRUNCATE_AT` and `UR_TOOL_BUILTIN_MAX_LINES` added to `Settings`.
`create_default_registry()` accepts `truncate_at` and `max_lines` params; `_make_registry`
passes them from settings. `functools.partial` binds the values onto `shell`, `read_file`,
and `http_get` at registry creation time so the LLM cannot override them.

### Fix B — `aiofiles` lazy import

`builtin.py` originally imported `aiofiles` at module level. Since `aiofiles` is in the
`[tools]` optional extra, this caused `ImportError` in `_make_registry`, silently disabling
all tools even though `shell` and `http_get` don't need it. Fixed by moving the import
inside `read_file` and `write_file`. Now the registry loads and tools are registered without
installing `[tools]`.

### Fix C — `tool_call` / `tool_result` StreamChunk kinds

`StreamChunk.kind` extended with `"tool_call"` and `"tool_result"`. The loop yields these
before/after each tool execution so the display layer can show progress.

### Fix D — Sequential live display

Original display accumulated reasoning + content together in a single `Group`. Replaced
with a commit-on-transition model:

1. Reasoning streams in the live area as dim text.
2. On `tool_call`: reasoning committed via `console.print` (persists), live cleared, `⚙`
   printed.
3. On `tool_result`: `🤖` preview printed, `content_acc` reset.
4. On first content token after reasoning: reasoning committed, live switches to `Markdown`.

Result: `<reasoning pt1>` → `⚙ call` → `🤖 result` → `<reasoning pt2>` → `<reply>`.

### Fix E — Ollama package replaces litellm for Ollama provider

**Root cause:** litellm 1.81.15 puts Ollama streaming tool calls into `delta.content` as
plain JSON text instead of `delta.tool_calls`. `has_tool_calls` was always `False` for
Ollama, so tools were never dispatched.

**Fix:** Switched the Ollama provider path to use the `ollama` Python package directly
(`ollama>=0.4` added to base deps). `OllamaCompletionStream` subclass reads
`chunk.message.tool_calls` (already-parsed dicts) and `chunk.message.thinking` natively.
litellm is still used for Gemini and all other providers.

### Fix F — Ollama message format conversion

The session stores tool call `arguments` as a JSON string (OpenAI wire format). The
`ollama` package's pydantic `Message` model requires `arguments` to be a `dict`. Added
`_to_ollama_messages()` helper that deserialises `arguments` strings back to dicts before
passing messages to `ollama.AsyncClient.chat()`.

### Fix G — `UR_MODEL_THINK` setting

`model_think: bool` added to `Settings` (env var `UR_MODEL_THINK`). Passed as
`think=self.settings.model_think or None` to `ollama.AsyncClient.chat()`. When `True`,
the Ollama package separates thinking tokens into `chunk.message.thinking`, which maps to
`StreamChunk(kind="reasoning")` and renders as dim text. When `False` (default), `think`
is omitted entirely so non-thinking models are unaffected.
