## Discussion: tool_call invoking `ur run` as a sub-agent

Explored whether a tool named `run_agent` could let the LLM spawn a sub-agent
by emitting a `tool_calls` JSON block. The host would execute `ur run <prompt>`
as a subprocess, capture stdout, and inject the result as a `tool` role message
before continuing the conversation.

### Prerequisites

Tool-call support requires changes that do not exist in Phase 3:

- **Tool-call parsing** — detect `tool_calls` array in provider response JSON
  instead of plain `content`; extract tool name and arguments.
- **Tool dispatch** — a registry mapping tool names to handlers.
- **Message continuity** — append a `tool` role result and loop back to
  `provider.complete()`.

These are Phase 4 responsibilities. Phase 3 is unaffected.

### Async vs synchronous

True async (non-blocking sub-agent while the chat loop keeps accepting input)
introduces ordering problems in message history and complicates the TUI.
Synchronous execution — block the loop, run `ur run` to completion, inject
result — is simpler and the spinner already covers the UX.

### Design sketch

```
tool name:     run_agent
arguments:     { "prompt": "<sub-task>" }
implementation: spawn  ur run "<prompt>"  as a subprocess
                capture stdout → tool result string
                append Message{"tool", result} to context
                call provider.complete() again
```

Each `ur run` invocation creates its own session row. SQLite WAL mode handles
concurrent access safely. Only `result.response` is written to stdout (log goes
to file), so stdout capture is clean.

### Pros

- Clean isolation — sub-agent gets its own session ID and log entries.
- No in-process coupling; `ur` binary boundary keeps tools simple.
- Shares the same DB and encryption key naturally (correct behavior).
- Synchronous model fits the existing spinner UX without new TUI complexity.

### Cons / risks

| Risk | Mitigation |
|------|-----------|
| Infinite recursion (agent → run_agent → agent → …) | Depth counter via env var or CLI flag |
| Sub-agent latency blocks parent turn | Acceptable for sync; spinner provides feedback |
| Tool-call parsing adds provider coupling | Isolate in a `parse_tool_calls()` helper, keep `Provider::complete()` unchanged |

### Conclusion

Feasible and architecturally sound. Deferred to Phase 4 for the `run_agent`
tool implementation. Phase 3.1 handles the prerequisite flag redesign on
`ur run` so Phase 4 can wire sub-agent invocation without touching the CLI
layer again.

## Design decision: tool plugin configuration vs selection

Resolved the question of `--tools=<path>` (previously planned in Phase 4).

Split into two distinct concerns:
- **Configuration** (persistent): `$root/tools/tools.json` — which tools exist,
  their settings and constraints. Written once by the user, read at startup.
- **Selection** (ephemeral): `--allow`/`--deny`/`--no-tools` CLI flags — which
  tools are active for a specific invocation.

`--tools=<path>` conflated both and is dropped. This matches the MCP pattern
(tool servers configured persistently, not re-specified per call).

Plans updated: `docs/plan/phase3.1.md` created, `docs/plan/phase4.md` and
`README.md` updated to remove `--tools`.

## TUI layout finalised (4 tabs)

Settled the `FtxuiTui` tab structure:

| Tab | Content | Phase |
|-----|---------|-------|
| Session | Chat history + input + slash autocomplete + status footer | 3 |
| System Prompt | Multi-line editor, mid-session effect, save/load file | 3 |
| Tools | Checkbox list (placeholder → wired in Phase 4) | 3→4 |
| Options | ENV settings persisted to `.env` on save | 3 |

Key decisions:
- Slash command autocomplete only (not message content).
- System prompt edits take effect on next provider call within the session.
- Options changes write back to `.env`.
- `/compact` is a slash command (stub in Phase 3, full summarisation in Phase 5).
  Compact inserts a `[summary]` DB record so `--continue` honours it.
- Context manager dropped as a separate tab. Status footer in Session tab shows
  `context: N/ctx_len tokens` — N from `usage.prompt_tokens` in the API
  response, ctx_len from `GET /props` (llama.cpp `n_ctx`) via a new
  `Provider::server_info()` call at session start.

## TUI rendering fixes

Several rendering bugs found and fixed during Phase 3 chat testing:

- **Text overflow** — `paragraph()` inside a `Renderer` did not get a width
  constraint, so long lines ran off screen. Fixed by adding a `render_text()`
  helper that splits on `\n` with `std::getline` and wraps each line in
  `paragraph()`, and by applying `xflex` (horizontal flex only) instead of
  `flex` to message vboxes.
- **`\n` not rendered as newlines** — `paragraph()` treats newlines as
  whitespace. `render_text()` splits the string on `\n` first, producing one
  `paragraph()` per line.
- **Vertical gap between messages** — `flex` on the message vbox caused it to
  expand vertically to fill remaining history area. Replaced with `xflex`.
- **Label colours** — `user:` label rendered in Cyan, `assistant:` in Green.
- **Mouse scrolling** — `vscroll_indicator | frame` on a rendered Element
  provides visual scroll only; mouse wheel events require Component-level
  handling. Replaced with a custom `ScrollerBase` (`ComponentBase` subclass)
  that intercepts `WheelUp`/`WheelDown` and arrow/page/home/end keys, and uses
  `yframe` + a virtual focused element to control the viewport. Children (e.g.
  `Collapsible`) receive events first via `ComponentBase::OnEvent`; scroll
  handling only fires if no child consumed the event. Auto-scrolls to bottom on
  each new message via `ScrollToEnd()`.

## Markdown rendering plan

Decided to use **md4c** (C library, ~50 KB) for rendering LLM markdown output
in the TUI. md4c parses CommonMark; the renderer will walk the AST and emit
ftxui elements (bold, inline code, code blocks, headers, lists). Deferred to
after core Phase 3 stability work.

## Provider interface extended

`provider.hpp` gains `CompletionResult` (content + token counts) and
`ServerInfo` (context_length). `Provider::complete()` now returns
`CompletionResult`; `Provider::server_info()` fetches server capabilities.
`HttpProvider` implements `server_info()` via `GET /props` (llama.cpp-specific,
fails silently for other servers). `runner.cpp` and test mocks updated
accordingly.
