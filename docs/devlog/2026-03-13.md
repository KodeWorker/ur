### Session title — `/title` command + unique index

**Design**: sessions are untitled by default (stored as SQL `NULL`, not `""`).
A unique partial index enforces uniqueness only over non-NULL, non-empty titles:

```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_session_title
    ON session(title) WHERE title IS NOT NULL AND title != '';
```

`insert_session` binds `NULL` when `title` is empty so multiple untitled
sessions don't conflict. `runner.cpp` was using `prompt.substr(0, 60)` as the
title — changed to `""` (untitled).

New Database methods:

| Method | Purpose |
|--------|---------|
| `update_session_title(id, title)` | Set or clear a session title |
| `find_session_by_title(title)` | Return ID for exact title match; throws if not found |
| `find_session_by_id_prefix(prefix)` | Return ID for unique prefix match; throws if 0 or >1 matches |

`/title <text>` slash command added to `Chat::run()` — calls
`update_session_title` and prints confirmation. Bare `/title` (no arg) shows
usage. `/title` added to the tab-autocomplete list in `FtxuiTui`.

### `--continue` accepts title or short ID prefix

Resolution order in `cmd_chat`:
1. Exact session ID (`session_exists`)
2. ID prefix match (`find_session_by_id_prefix`) — e.g. `--continue=d21e78`
3. Title match (`find_session_by_title`) — e.g. `--continue="my chat"`

### `--continue` history replay fix (TUI)

Prior to this fix, `Chat::run()` called `tui.print_user/print_response/
print_reasoning` before `read_input()` started the ftxui event loop thread.
These calls went to `screen.Post(...)` which queues tasks for the event loop
— but with the loop not yet running, the tasks were processed only after the
first render, and on some ftxui builds were silently dropped.

Fix: `Impl::post_or_direct(fn)` helper added to `FtxuiTui::Impl`. Before
the loop thread starts (`!loop_thread.joinable()`), the lambda executes
directly and synchronously on the main thread — components are in
`history_container` before the first render. Once the loop is running, the
normal `screen.Post` path is used.

All `print_user`, `print_response`, `print_reasoning`, and `print_error`
methods updated to use `post_or_direct`.

### TUI scroller — scroll-to-top bug on input focus

**Root cause**: `ScrollerBase::Render()` used a conditional anchor decorator:

```cpp
auto focused = Focused() ? ftxui::focus : ftxui::select;
```

`ftxui::yframe` only scrolls to children decorated with `ftxui::focus`. When
keyboard focus moved to the input component, the scroller was no longer
`Focused()`, so the anchor switched to `ftxui::select` and `yframe` snapped
to the top of the content on every render triggered by a keypress.

**Fix**: always use `ftxui::focus` as the anchor element in the `dbox` virtual
cursor. The Element-level `ftxui::focus` decorator is a rendering hint for
`yframe` — it is independent of Component keyboard focus (Tab navigation) and
does not interfere with it. `selected_` continues to control which row
`yframe` scrolls to regardless of which component holds keyboard focus.

**Removed**: `FocusWatcherBase` component that attempted to trigger
`ScrollToEnd()` on input focus. The approach was unreliable because
`Container::Vertical` renders children top-to-bottom, so the scroller always
rendered before the watcher fired, and `Event::Custom` used to force a second
render caused `was_focused_` to reset, creating a scroll loop that could land
on `selected_=0` (top) when `size_` was briefly zero during layout.
